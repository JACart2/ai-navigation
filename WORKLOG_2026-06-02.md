# Worklog: 2026-06-02

Repository: `/root/dev_ws/src/ai-navigation`

Branch: `Localization_fix`

## Goal

Debug and improve the cart localization/RViz workflow, especially live Velodyne visibility in RViz and TF stability between `map`, `odom`, `base_link`, and `velodyne`.

## Main Symptoms Investigated

- RViz showed the map, but live Velodyne points were not rendering reliably.
- RViz reported transform problems when trying to transform Velodyne data into fixed frame `map`.
- Runtime logs showed repeated TF timing warnings:

```text
Could not get transform base_link to odom:
Lookup would require extrapolation into the future
```

- The rear ZED camera also produced a startup error in one run:

```text
zed_rear.zed_node_1: Error opening camera: INVALID FUNCTION CALL
```

## Current TF Theory

RViz needs this transform chain to display live Velodyne points in fixed frame `map`:

```text
map -> odom -> base_link -> velodyne
```

The debugging focused on reducing TF ownership conflicts and timing gaps:

- Avoid ZED and lidar localization both publishing global TF.
- Keep `base_link -> velodyne` static and stable.
- Keep ZED odometry available as topics without letting it fight the localization TF tree.
- Reduce failures caused by point cloud timestamps being slightly ahead of the newest odom TF.

## 2026-06-03 Follow-Up: Cart Motion And Localization While Driving

### Goal Publish But Cart Did Not Move

The navigation command chain is:

```text
/clicked_point -> global_planner -> /global_path -> local_planner -> /nav_cmd -> motor_endpoint -> Arduino serial
```

Most likely blocker found:

- `cart_launch` includes `motor_control/launch/motor.launch.py` but does not currently forward an `arduino_port` launch argument.
- `motor.launch.py` defaults to `/dev/ttyUSB0`.
- The motor control udev rule creates the Arduino symlink as `/dev/ttyUSB9`.
- If the Arduino is actually on `/dev/ttyUSB9`, `motor_endpoint` will keep retrying serial and return before writing packets, even if `/nav_cmd` is published.

Relevant files:

```text
cart_control/cart_launch/launch/autonomous_launcher.launch.py
motor_control/launch/motor.launch.py
motor_control/resource/99-usb-serial.rules
motor_control/motor_control/motor_endpoint.py
```

Runtime checks:

```bash
ros2 topic echo /heartbeat --once
ros2 topic echo /nav_cmd --once
```

Possible next fix: expose `arduino_port` in `autonomous_launcher.launch.py` and forward it to `motor.launch.py`, then launch with `arduino_port:=/dev/ttyUSB9`.

### Localization Loss While Driving

Root symptom while driving:

```text
Could not get transform base_link to odom:
Lookup would require extrapolation into the future
```

Cause theory:

- Velodyne point cloud timestamps can run ahead of the newest ZED odom TF by roughly 20-100 ms.
- `lidar_localization_ros2` needs an odom/base transform at the scan timestamp.
- When that lookup fails, localization stops publishing a usable `map -> odom` / pose update path, which looks like localization dropping out while driving.

Fix applied:

- ZED now owns `odom -> zed_front_camera_link`.
- A static transform now completes `zed_front_camera_link -> base_link`.
- Lidar localization owns `map -> odom`.
- ZED `map -> odom` remains disabled so it does not fight lidar localization.
- ZED TF is future-dated by `0.15` seconds to cover the measured timestamp skew.

Files changed:

```text
cart_control/localization_launch/config/common.yaml
  pos_tracking.transform_time_offset: 0.15

cart_control/localization_launch/param/localization.yaml
  use_odom: true
  use_twist_prediction: true
  max_twist_prediction_dt: 0.35
  enable_map_odom_tf: true

cart_control/localization_launch/launch/zed_multi_camera.launch.py
  publish_tf: true
  publish_map_tf: false
  passes localization_launch/config/common.yaml as config_path

cart_control/localization_launch/launch/cameras.launch.py
  static TF direction changed to zed_front_camera_link -> base_link
```
